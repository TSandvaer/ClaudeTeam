## Patterns + anti-patterns to internalize

- **PATTERN — Staged-rule authoring + apply-on-direct-request.** When a real in-session incident surfaces a global-orchestrator-discipline gap, author the rule as a `team/log/proposed-global-rule-<name>-YYYY-MM-DD.md` staged doc the same session. Sponsor reads + applies, OR when sponsor's immediate-prior message is the authorization, orchestrator direct-applies via path (b) of the classifier-blocks workaround. After application, move the staged doc to `team/log/applied/` with an `applied-rule-` rename for audit trail. Validated 5× this M3.
- **PATTERN — Sponsor's pointed question = candidate trigger for a new global rule.** Treat every pointed sponsor question ("are you stuck", "can this be prevented", "look at what you produced") as a candidate trigger for a class-level orchestration failure that the orchestrator's own introspection mechanisms did not catch. The 1:1 mapping observed in M3 (4 pointed questions → 4 of the 5 new rules) suggests the asymmetry is durable, not coincidental.
- **PATTERN — NITs-absorption-into-DOWNSTREAM-DISPATCH-BRIEF.** When a follow-up NIT informs the scope of the NEXT ticket already in the dispatch queue, the orchestrator absorbs the NIT into the downstream brief instead of filing as a separate NITs ticket. Reduces churn; downstream brief gains a "this NIT is absorbed: <description>" cross-reference. Done 1× this M3 (M3-01 NIT #3 → M3-02 dispatch brief).
- **PATTERN — Reconciliation rebase via sed-rename when vocabulary conflicts are scoped to identifiers.** When parallel PRs produce non-mergeable conflicts that boil down to "same shape, different names," the recovery is mechanical: rebase onto main, `git checkout --ours` for files with canonical-vocab conflicts (take the merged-first PR's names), sed-rename the second PR's branch to match, drop redundant type defs that the merged-first PR already provides, force-push, wait for CI. Recovery cost ~5-10 min vs. ~half a dispatch cycle for re-implementing the whole second PR. Validated on M3-10 PR #48.
- **ANTI-PATTERN — Shape-contract-without-identifier-names in parallel dispatch.** When dispatching multiple agents in parallel to work on different surfaces of a SHARED CONCEPT (new type, event shape, wire-format field, guard function), specifying only SHAPE (`{personaName, count, instances}`) and DIRECTIONALITY (X exports → Y imports) is insufficient. Each agent independently picks plausible identifier names; cross-review misses the divergence; second-merging PR is non-mergeable. Prevention: Pattern A (sequence dispatches with type-author first) OR Pattern B (explicit Vocabulary contract block naming all 5 identifiers — type / union / guard / discriminator / export-site). New rule "Parallel-agent shared-concept vocabulary discipline."
- **ANTI-PATTERN — Bash-run-in-background as sole wake signal for "wake me when X is ready" patterns.** Documented Monitor-tool pattern, but in practice the completion `<task-notification>` does not reliably surface when the bash exits with zero stdout output. Pair with `ScheduleWakeup` as belt-and-suspenders. Cron is too coarse (15-min worst case) for fast-completing time-critical events. New rule "Orchestrator wake-signal discipline."
- **ANTI-PATTERN — Acknowledged-then-regressed main-thread bloat.** Sponsor flags bloat → orchestrator acknowledges + applies fix for the current turn → next session regresses to the same bloat patterns because the discipline was per-session, not persistent. The acknowledgment-then-regress cycle is itself evidence that bloat discipline needs global-rule status, not session-local sponsor reminders. New rule "Orchestrator main-thread bloat discipline" (10 specific patterns).
- **ANTI-PATTERN — Trusting prior mental model of in-flight agent state on cron-tick wake-up.** Orchestrator's reflex when cron fires is to restate the prior assumption ("still waiting on Felix"). Background Agent tasks can die silently (session-ended, crashed, hung on permission prompt, completed-with-lost-notification). The cost of an independent verification check is ~5 seconds (`TaskOutput` + `git fetch` + `gh pr list`); the cost of stale-state confidence is sponsor blocking on a dead agent for an entire cron cycle. New rule "Background-agent staleness verification."
- **ANTI-PATTERN — Authoring follow-up dispatches from review comment text without re-reading cited code.** Orchestrator wrote `86c9ybrk0` dispatch from Maya's M3-04 review comment text alone, without opening the cited webview-boot file. Felix initially fixed the wrong layer (parser instead of webview-boot dev-fixture gating). Sponsor caught it mid-dispatch. Prevention: when transcribing a peer-reviewer's NIT into a follow-up dispatch brief, the orchestrator opens the cited `file:line` ref to verify scope interpretation — comment text alone is insufficient when the NIT involves cross-layer terminology.
