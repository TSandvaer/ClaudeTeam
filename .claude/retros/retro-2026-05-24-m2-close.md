## Surprising findings

- **Auto-decide reversal rate is well below the 5% calibration floor.** 10/10 = 0% reversal rate across M2's auto-merges. Per orch-autonomy rule 6.5, "<5% (almost nothing reversed) → orchestrator is being too cautious; surface fewer items, auto-decide more." Six auto-merge entries plus three NITs auto-dispatches plus one P0 auto-merge — sponsor has effectively delegated this entire class. **Implication:** more classes should be promoted to rule 6.6 auto-decide territory. Candidates from M2 patterns: (1) NITs-ticket-creation from APPROVE_WITH_NITS review comments (sponsor never disagreed with the orchestrator's filed scope this cycle); (2) `clickup-pending.md` collision recovery via `git checkout --ours` for log-only conflicts (already in failure-mode prevention, but elevation to explicit auto-decide class would close the audit gap); (3) NITs-absorption-into-downstream-ticket calls when the downstream consumer is already scheduled and the ticket files overlap (Path Y was sponsor-signed but the heuristic is reusable). See "Durable lessons promoted" for the recommendation.
- **The `mcp__clickup__update_task` classifier denial was non-deterministic across same-session boundaries.** Initial hypothesis (after the first hit on `86c9y7y9z`, 2026-05-24) was "tickets created in a *prior* session." Second hit (`86c9y9q6h`, M2-06) invalidated that hypothesis — the ticket was created in the *same* Claude-process session ~20 min earlier, after a cron-tick boundary. True trigger remains unknown (time-elapsed, cron-tick context reload, classifier sampling). The `settings.json` allow-list bypass is the right fix; investigating the classifier's true behavior is out of scope for this team but worth raising to Anthropic via the harness-feedback channel.
- **Path Y was orchestrator-recommended in STATE.md "Wave 2 ready to dispatch" + sponsor-confirmed within one round.** The orchestrator articulated both paths with clear cost/benefit framing; sponsor's pick was unsurprising. This is the structure to emulate when a sponsor decision is needed — surface both paths with named tradeoffs, then make the recommendation. Cheaper than "Path A or Path B?" with no orientation, and faster than the orchestrator silently picking + having to defend later.
- **M2-08's `xvfb-run` Layer-3 setup on CI Ubuntu worked first try.** No flakes, no display-related failures across the 14 tests. This is non-obvious — Layer-3 in CI is famously fragile in many projects. Sage's M2-08 PR set the pattern cleanly; Layer-3 expansion in M3+ inherits a known-good pipeline.
- **The sub-agent GUI gap is permanent harness behavior — applies to every orchestrated project on this machine.** ClaudeTeam happens to need it most visibly because webview-smoke is in CLAUDE.md hard rule #3. Sibling projects with web frontends or desktop apps will hit the same wall. Worth porting the testing-strategy.md § "Sub-agent GUI gap" reframe into `create-orchestration-project` skill's `template/` for future projects.

## Patterns + anti-patterns to internalize

- **PATTERN — APPROVE_WITH_NITS + follow-up ticket (or PR comment for single-line NITs).** Used 7× this M2. The discriminating call is: 2+ NITs → file a ticket; 1 single-line NIT → leave as PR comment, no ticket overhead. Validated by every PR that used it (no NITs lost, no over-ticketing).
- **PATTERN — Sponsor-pick-the-path framing.** When a coordination call has two reasonable options, the orchestrator articulates both with cost/benefit + a recommendation. Sponsor responds in one round. Validated on Path Y, post-merge-screenshot-deferral, `settings.json` allow-list authorization.
- **PATTERN — Sub-agent GUI gap deferral (AC(a) load-bearing, AC(b-d) post-merge).** Documented in `.claude/docs/testing-strategy.md` § "Sub-agent GUI gap". Applies whenever both author and reviewer are sub-agents and data-plane smoke is performable. NOT a license to skip screenshots when a human is in the loop.
- **PATTERN — Live-runTick smoke as data-plane verification.** Sub-agents CAN run the production `runTick()` against real `~/.claude/` data and cite materialized team/agent objects. This is the load-bearing pre-merge verification for any extension PR that exercises the parse → match → state-emit pipeline. Validated by Maya's M2-06 review (real `claudeteam-alpha` team materialized with Felix + Maya tiles + 7 background agents).
- **PATTERN — Path Y / NITs-absorption-into-downstream.** When a follow-up NITs ticket and a downstream ticket touch the same files (`messageBus.ts`, `main.ts` in M2's case), combine into one larger PR rather than ship two sequential. The downstream consumer benefits from the NIT fixes; the larger PR is more economical than two cycles. Applicable whenever the NITs ticket has no independent timeline pressure.
- **PATTERN — Permission-rule allow-list for recurring classifier denials.** When the auto-mode classifier non-deterministically denies an MCP tool the orchestrator structurally needs (status flips, doc writes), add to `.claude/settings.json` `permissions.allow` rather than re-asking sponsor per occurrence. Sponsor signs off once; classifier is bypassed permanently for that tool + project.
- **ANTI-PATTERN — Chain of deferred manual validations.** M2-01 deferred screenshots to M2-05 (placeholder-PR exception). M2-06 deferred screenshots to sponsor post-merge (sub-agent GUI gap). Neither was wrong individually, but the chain meant the `.vsix` install path was never exercised by a human until Layer-3 caught the activation bug. Prevention: when a PR is the FIRST surface of a runtime path (here, the installable `.vsix`), the screenshot/install validation should bind to THAT PR — even if the visible UI is a placeholder, the install path is the load-bearing test.
- **ANTI-PATTERN — Authoring test plans without checking who can execute them.** Sage's M2-07 test plan authored a webview-smoke gate that no sub-agent could execute pre-merge. Caught at M2-06 dispatch time, not at M2-07 authoring time. Prevention: test-plan PRs should explicitly map each manual-verification AC to a role + runtime capability (human / sub-agent / Layer-3-automated), and flag any AC that has no executor in the in-loop roster.
- **ANTI-PATTERN — Orchestrator narration of merge decisions in main thread.** Every auto-merge cycle this M2 produced a 10-20-line block in the main conversation describing the decision the orchestrator just made + filed in `decisions-while-away.md`. The log file IS the audit record; the main-thread narration is duplication. Prevention: post a one-line acknowledgment in main thread ("PR #N auto-merged, decision logged"); the detail stays in the log file.
