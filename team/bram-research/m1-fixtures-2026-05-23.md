## Schema divergence summary (for data-sources.md update consideration)

The real on-disk meta.json schema at v2.1.145 has THREE variants, not two:

| Variant | `agentType` value | `name` field | `toolUseId` | Observed in |
|---|---|---|---|---|
| Old (v2.1.119) | persona name (e.g., "devon") | absent | absent | NOT FOUND in scope |
| New-generic (v2.1.145) | "general-purpose" or "Explore" | absent (0/10 observed) | present | Sessions 5652d46e, 7b53d0ee |
| New-persona (v2.1.145) | persona name (e.g., "bram", "felix") | absent | present | Session 5652d46e |

The "new-persona" variant is undocumented. It appears when the orchestrator dispatches a named persona agent. The `toolUseId` is present (unlike the old schema), but `agentType` holds the persona name (like the old schema). The roster matcher must handle this: `agentType_equals: "felix"` will match both old-schema and new-persona agents correctly.

**Decision draft:** Update `.claude/docs/data-sources.md` to document the "new-persona" variant (v2.1.145, persona name in `agentType`, `toolUseId` present, no `name`).

## Implications for ClaudeTeam

- **Felix's meta.json parser (M1-05):** must handle three schema variants, not two. Feature-detect: if `toolUseId` present AND `agentType` is a persona name → "new-persona" variant; if `toolUseId` absent → old schema; if `agentType == "general-purpose"` or other engine type → "new-generic". The `name` field, if present at all, may be `null` explicitly — handle both `undefined` and `null`.
- **subagent-finished.jsonl:** the synthesized closing line (line 7) is the only way to exercise `stop_reason: "end_turn"` in a fixture; in real captures that field is never the last line. Felix should be aware the "finished" detection logic based on subagent JSONL content alone is unreliable — the real signal is presence of `tool_result` in the parent JSONL matching `toolUseId`.
