# M3 Prior-Art: Settings-UI Patterns + Global FS Watching ‚Äî 2026-05-24

## Question

Two design questions for M3 (roster config):

1. For the sponsor-curated YAML roster at `~/.claudeteam/teams.yaml`, should ClaudeTeam use (a) the native VS Code Settings UI via `contributes.configuration` + `vscode.workspace.getConfiguration`, or (b) a custom webview-form roster editor in the existing dashboard webview?

2. Can `vscode.workspace.createFileSystemWatcher` watch `~/.claudeteam/teams.yaml` ‚Äî a file outside any VS Code workspace folder? What are the caveats and what fallbacks exist?

---

## Answer ‚Äî Question 1 (Settings UI vs Custom Webview Form)

**Approach (a) native Settings UI is a poor fit for our roster schema.** The roster YAML contains nested arrays of objects (`agents[].matchPatterns[]`, `agents[].color`, etc.), and the VS Code Settings UI explicitly falls back to "Edit in settings.json" for any property that is array-of-objects ‚Äî the native UI does not render editable grids for that shape. The correct approach for M3 is to treat the sponsor as a YAML file editor: load `~/.claudeteam/teams.yaml` directly, validate with Zod, surface errors in the dashboard, and add a `claudeteam.openRoster` command that opens the file in VS Code's native editor. The native Settings UI is correct for the three scalar settings already declared (`rosterPath`, `pollIntervalMs`, `showBackgroundCount`); it should not be stretched to carry the roster data.

**Approach (b) custom webview-form is out of scope for V1.** It requires full CRUD UI, validation UX, and write-back logic ‚Äî disproportionate complexity for a 1-2 day milestone on a sponsor who is comfortable editing YAML directly.

---

## Answer ‚Äî Question 2 (createFileSystemWatcher outside workspace)

**`vscode.workspace.createFileSystemWatcher` CAN watch `~/.claudeteam/teams.yaml` outside the workspace**, using `new vscode.RelativePattern(vscode.Uri.file(dir), '*.yaml')` where `dir` is the absolute path to `~/.claudeteam/`. This feature was added in VS Code 1.64 (January 2022) and is within our `^1.85.0` engine floor. Key caveats: (1) watching with a plain glob string (not `RelativePattern`) filters out all out-of-workspace events silently; (2) if the watched path does not yet exist at watcher-create time, VS Code falls back to a 5-second `fs.watchFile` poll until the path appears; (3) a confirmed bug (issue #164925, opened Oct 2022, resolution status unverified) reports that using a complete filename without wildcards may not fire on all VS Code builds ‚Äî the workaround is to use `*.yaml` as the glob pattern rather than `teams.yaml` literally; (4) recursive-watching outside workspace with `**` patterns has a known double-fire bug on some builds (issue #163352, closed-not-planned); our single-level `*.yaml` pattern avoids the recursive path entirely.

**M2's decision to avoid chokidar still holds.** Node `fs.watch` would require a manual poll fallback (as Pixel Agents does), adding complexity. The VS Code watcher is zero-dep, lifecycle-integrated, and sufficient.

## Evidence

### Question 1 ‚Äî Settings UI patterns

#### What the VS Code Settings UI can and cannot render

- **Source: VS Code Contribution Points docs** (`https://code.visualstudio.com/api/references/contribution-points#contributes.configuration`) ‚Äî key quote verified via fetch: "If an `object` or `array` type setting can also contain other types like nested objects, arrays, or null, then the value won't be rendered in the settings UI and can only be modified by editing the JSON directly. Users will see a link to **Edit in settings.json**."
- **Supported in the visual UI:** simple arrays of `number`, `string`, or `boolean` (render as editable lists); flat objects whose properties are all `string`/`number`/`integer`/`boolean` (render as key-value grids). Our roster does not fit either shape.
- **What our schema looks like:** `teams[].members[].match[]` is a list of single-key objects. `teams[].members[].color` is a string. This is an array of objects with nested arrays ‚Äî it falls squarely into the "Edit in settings.json" bucket.

#### Canonical real-world example: Claude Code extension `claudeCode.environmentVariables`

The Claude Code VS Code extension (`anthropic.claude-code-2.1.145-win32-x64`) is the closest real-world prior art for an array-of-objects setting. Its `package.json` at `C:\Users\538252\.vscode\extensions\anthropic.claude-code-2.1.145-win32-x64\package.json` declares:

```json
"claudeCode.environmentVariables": {
  "type": "array",
  "items": {
    "type": "object",
    "properties": {
      "name": { "type": "string", "description": "Environment variable name" },
      "value": { "type": "string", "description": "Environment variable value" }
    },
    "required": ["name", "value"]
  },
  "default": [],
  "description": "Environment variables to set when launching Claude..."
}
```

This is a simpler array-of-objects than our roster (only two flat fields). Despite that simplicity, the Claude Code extension's own description field notes: "Prefer setting environment variables in Claude's settings.json." The extension explicitly redirects users to JSON editing rather than the native UI. How the extension reads the value (verified from `extension.js`): `vscode.workspace.getConfiguration("claudeCode").get("environmentVariables")` ‚Äî a standard `getConfiguration` read. The array arrives as a plain JS array of objects after JSON deserialization.

- **Path verified:** `C:\Users\538252\.vscode\extensions\anthropic.claude-code-2.1.145-win32-x64\package.json` (read this session).
- **extension.js code pattern verified:** grep on `extension.js` confirmed: `getConfiguration("claudeCode")` + `.get("environmentVariables")` ‚Üí result iterated as `for(let N of K) if(N.name) V[N.name]=N.value||""`

#### VS Code UX guidelines position (verified)

VS Code UX guidelines (`https://code.visualstudio.com/api/ux-guidelines/settings`, fetched this session) state: "Don't: Create your own settings page/webview." The guideline is about keeping configuration discoverable via the Settings UI. However, this rule applies to settings that ARE representable in the Settings UI. For complex structured data like our roster, the guidelines assume the user falls back to `settings.json` (or in our case, the YAML file directly).

#### What approach (a) would actually look like for the roster

If ClaudeTeam declared `claudeteam.teams` as a configuration property with type `array` and items of type `object` (with `members[]`, `match[]` sub-arrays), the Settings UI would show: "Edit in settings.json" link only ‚Äî no form fields. The value would be stored as JSON in `settings.json` (not YAML), which contradicts the roster schema (`~/.claudeteam/teams.yaml`). YAML round-trips through JSON settings would require custom serialization and defeat the purpose of the YAML file format.

**Write-back scope issue (User vs Workspace):** `vscode.workspace.getConfiguration` writes to User scope (global) or Workspace scope (project `.vscode/settings.json`). The global `teams.yaml` file is intentionally placed at `~/.claudeteam/` to be independent of any VS Code workspace. Forcing the roster through VS Code settings creates a coupling: the roster would only exist in VS Code's settings store, not as a portable YAML file inspectable outside VS Code. This conflicts with the documented architecture (`roster-matching.md` ¬ßConfig locations, `.claude/docs/data-sources.md` ¬ßroster-matching.md).

#### Approach (b) ‚Äî custom webview form

A custom webview form (CRUD UI in the existing dashboard) would require:
- Full form rendering for team/member add/edit/delete actions.
- Validation UX (field-level errors in the webview).
- Write-back to `~/.claudeteam/teams.yaml` via the extension host (host reads ‚Üí validates ‚Üí writes YAML via `js-yaml.dump`).
- A new message type in `src/shared/messages.ts` (e.g. `roster:edit`/`roster:save`).

This is disproportionate for M3 scope. The sponsor edits YAML directly and is comfortable with that workflow ‚Äî confirmed by the fact that `C:\Users\538252\.claudeteam\teams.yaml` already exists with a correct hand-authored roster (read this session). A "open file in VS Code editor" command is sufficient for M3.

#### What the M3 implementer actually needs

The already-scaffolded `claudeteam.openRoster` command (already in `package.json` `contributes.commands` ‚Äî verified at `C:\Trunk\PRIVATE\ClaudeTeam\package.json` line 54) is the correct UI surface for roster editing. Implementation: `vscode.workspace.openTextDocument(rosterPath).then(doc => vscode.window.showTextDocument(doc))`. This opens the YAML in VS Code's native editor with YAML syntax highlighting (if the user has a YAML extension installed). Validation errors from the Zod schema are surfaced in the dashboard's error chip (already planned in `roster-matching.md` ¬ßLoader edge cases). No custom webview form needed.

### Question 2 ‚Äî createFileSystemWatcher outside workspace root

#### VS Code 1.64 feature introduction (authoritative)

- **Source: VS Code 1.64 release notes** (fetched `https://code.visualstudio.com/updates/v1_64`, confirmed by search results): "The existing `vscode.workspace.createFileSystemWatcher` API was improved to allow you to pass in any file or folder path for file watching, even if it is outside of the workspace."
- **Example from release notes (exact):** `vscode.workspace.createFileSystemWatcher(new vscode.RelativePattern(vscode.Uri.file(<path to folder outside workspace>), '*.js'))` ‚Äî the same pattern applies with `'*.yaml'` for our use case.
- **Engine floor 1.85 covers this:** 1.85 > 1.64 by 21 releases. Confirmed in prior research note `team/bram-research/m2-vscode-prior-art-2026-05-23.md` ¬ß"File-watcher approach".

#### The string-pattern trap (verified behavior)

If you call `createFileSystemWatcher('**/*.yaml')` (a plain string), VS Code 1.64+ explicitly changed behavior: "A file watcher that is instructed with just a glob pattern (for example, `vscode.workspace.createFileSystemWatcher('**')`) will no longer receive events for files that changed outside of the workspace." Plain string patterns are workspace-scoped. **You MUST use `RelativePattern` with a `Uri.file` base for out-of-workspace watching.**

Correct pattern for our case:
```typescript
const watchDir = path.dirname(rosterPath); // e.g. C:\Users\538252\.claudeteam
const watcher = vscode.workspace.createFileSystemWatcher(
  new vscode.RelativePattern(vscode.Uri.file(watchDir), '*.yaml')
);
watcher.onDidChange(() => reloadRoster());
watcher.onDidCreate(() => reloadRoster());
watcher.onDidDelete(() => clearRoster());
context.subscriptions.push(watcher);
```

#### Watcher backend (verified from VS Code File Watcher Internals wiki)

- **Recursive paths (glob with `**`):** Uses `ParcelWatcher` (parcel-bundler/watcher library).
- **Non-recursive paths (no `**`):** Uses `NodeJSWatcherLibrary` (Node `fs.watch`).
- Our `*.yaml` pattern on `~/.claudeteam/` is a non-recursive watch ‚Üí uses `fs.watch` internally via the VS Code wrapper.
- **Path not yet existing:** Falls back to a `fs.watchFile` poll at **5-second intervals** until the path appears. This is the behavior when `~/.claudeteam/` does not yet exist. It means: if the sponsor has never created their roster file, the watcher will poll at 5s until the directory/file appears, then switch to `fs.watch`. This is acceptable for our use case (one file, low-frequency edits).

#### Known issues with single-file watching

- **Issue #164925** (opened Oct 2022): "FileSystemWatcher not firing when complete filename used without wildcards." The bug reports that `createFileSystemWatcher(new vscode.RelativePattern(dir, 'teams.yaml'))` ‚Äî using the literal filename ‚Äî may fail to fire on some VS Code builds. **Resolution status: unverified** (page content was truncated; I could not confirm whether this was fixed and in which version).
- **Mitigation:** Use `'*.yaml'` as the glob (not `'teams.yaml'` literally). If other YAML files appear in `~/.claudeteam/`, filter by URI in the handler. The M3 roster loader already handles the `rosterPath` config override ‚Äî the handler can check `uri.fsPath === rosterPath` before reloading.
- **Issue #163352** (double-fire on `onDidChange` for recursive watchers outside workspace, VS Code edit): Applies only to recursive patterns with `**`. Our non-recursive `*.yaml` pattern is not affected. Marked closed-not-planned.

#### Platform note (Windows)

- **Issue #172939** (Windows-specific recursive watcher bug with `FOLDER_NAME\**`): Reported on Windows 10 with VS Code 1.74‚Äì1.75. Applies only to recursive patterns. Our non-recursive `*.yaml` pattern does not trigger this path.
- The VS Code 1.64 release notes note that directory-detection improvements "now also work on Windows" ‚Äî no Windows-specific caveats were documented for non-recursive out-of-workspace watching.

#### Fallback alternatives (for completeness, not recommended)

| Approach | Works outside workspace | Deps | Lifecycle | Windows reliability |
|---|---|---|---|---|
| `createFileSystemWatcher` + `RelativePattern` | Yes (VS Code ‚â•1.64) | Zero | Disposable (context.subscriptions) | Good (non-recursive, no `**`) |
| Node `fs.watch` | Yes | Zero | Manual start/stop | Unreliable on some paths; event deduplication required |
| `fs.watchFile` (polling) | Yes | Zero | Manual start/stop + clearInterval | Reliable but always polling; 5s default |
| chokidar v4 | Yes | 1 dep | Manual | Works; uses `fs.watch` internally on Windows |

**chokidar ruled out:** M2-02 decision (`team/bram-research/m2-vscode-prior-art-2026-05-23.md` ¬ß"File-watcher approach recommendation") explicitly ruled out chokidar ‚Äî "zero-dep approach available." That decision still holds. Our use case (one file, low-frequency edits) does not justify a dependency.

**Node `fs.watch` ruled out:** Pixel Agents' own code shows them pairing `fs.watch` with a 2-second `setInterval` fallback because `fs.watch` alone is unreliable. We'd be reinventing the VS Code watcher's reliability layer. Not worth it when the VS Code watcher covers our case cleanly.

**`setInterval` + `stat` polling:** Already used for JSONL tailing. For the roster YAML, adding a 2-second mtime stat-check inside the existing poll loop is a viable zero-setup alternative if the VS Code watcher proves unreliable in testing. This is the lowest-risk fallback: one extra `fs.statSync(rosterPath).mtimeMs` check per tick.
